# Privacy & Egress Audit — rust-lsp-mcp

> Generated by a multi-agent sweep (5 investigation dimensions, adversarial
> verification, completeness critic). Threat model: the scanned project's
> source/docs (bind-mounted `/project`, and the Markdown corpus) must not be
> exfiltrated off-host by any dependency.

## 1. Bottom line

There is **no confirmed path by which any dependency itself transmits your
scanned source or doc corpus off-host** — every first-party dataflow, the local
ONNX embedding path, ChromaDB telemetry/tracing, and the LSP transport were
traced to local-only sinks. The **one real content-exfiltration exposure is
indirect**: rust-analyzer compiles and runs the scanned project's own `build.rs`
and proc-macros on the host at startup (default-on), and because the container
ships with **no network isolation and no `CARGO_NET_OFFLINE`**, that untrusted
code has unrestricted outbound network and full read access to `/project`. So
content can leave the host **only if the scanned project (or one of its crates)
is malicious** — which is precisely this tool's "point it at any Rust project"
use case. Everything else (posthog, OTLP, huggingface stack, kubernetes/HTTP-
server deps) is either a no-op, off-by-default, or unreachable dead code.

## 2. Content-egress risks (highest priority)

### 2.1 rust-analyzer executes `build.rs` / proc-macros / `cargo check` from the scanned project — **HIGH**

- **Data at risk:** scanned Rust source (full read access via the executed code).
- **Verdict:** confirmed, reachable at runtime, `possible-content-egress`.
- **Evidence:** multilspy's bundled `initialize_params.json` enables the
  code-executing features — `cargo.buildScripts.enable=true`,
  `procMacro.enable=true`, `checkOnSave=true` / `check.command="check"`,
  `cachePriming.enable=true`. First-party `PatchedRustAnalyzer._get_initialize_params`
  (analyzer.py:122–142) overrides **only** `capabilities.general.positionEncodings`,
  leaving every code-exec setting on. The analyzer starts automatically on
  FastMCP boot, so build-script/proc-macro code from `/project` is compiled and
  executed on the host during startup indexing. The `:ro` bind mount blocks
  writes but **not reads**.
- **Fix:** (1) run with `--network none` or a deny-by-default egress firewall;
  (2) and/or override `initializationOptions` to disable `buildScripts`,
  `procMacro`, `checkOnSave` (accepting reduced type resolution); (3) at minimum,
  document that indexing runs untrusted code and require network isolation.

### 2.2 No cargo/rust-analyzer network isolation — the amplifier — **MEDIUM**

- **Data at risk:** scanned Rust source.
- **Verdict:** confirmed, reachable at runtime, `confirmed-content-egress`
  (path is default-on and unimpeded; realization contingent on malicious input).
- **Evidence:** zero `CARGO_NET_OFFLINE`, `--offline`, `--network none`,
  `network_mode`, or `seccomp` across `Dockerfile`, `.devcontainer/`, `README.md`,
  `src/`. cargo check / cargo metadata fetch deps over the network at runtime,
  and executed `build.rs`/proc-macro code can read every file under `/project`
  and POST it anywhere.
- **Fix:** (1) `CARGO_NET_OFFLINE=true` (needs vendored deps); (2) `--network none`
  or allowlist egress proxy; (3) integration test: index a project whose
  `build.rs` attempts an outbound connection, assert it is blocked.

> Cargo pulling crates.io is inbound fetch, not exfil by itself. The exfil
> primitive is build-script/proc-macro execution (2.1); network isolation (2.2)
> is the control that neutralizes it.

## 3. Metadata / telemetry egress (lower priority — all inert today)

- **ChromaDB product telemetry (posthog):** `Posthog.capture()` body is literally
  `pass` in chromadb 1.5.9; the `posthog` package is **not installed** (absent
  from `uv.lock`). Reachable but does nothing. Mitigations in place and correct
  (`Settings(anonymized_telemetry=False)` + `ANONYMIZED_TELEMETRY` /
  `SCARF_NO_ANALYTICS` / `DO_NOT_TRACK` env). **Keep them; re-verify on upgrade.**
- **ChromaDB OpenTelemetry / OTLP:** `chroma_otel_granularity` defaults to `None`;
  `otel_init()` returns before constructing any exporter — `tracer` stays `None`.
  Standard `OTEL_EXPORTER_OTLP_*` env does not enable it (chroma reads its own
  `CHROMA_OTEL_*`). Unreachable dead code.
- **Logging:** no network log sinks; records go to stderr. Log sites emit paths,
  counts, symbol names, positions, and the user query string — never `.rs` bodies
  or markdown chunk text.

## 4. Install/build-time & inbound-only network (not content leaks)

- **ChromaDB ONNX embedding model download (runtime, inbound) — LOW.** Default
  EF `ONNXMiniLM_L6_V2` downloads `all-MiniLM-L6-v2/onnx.tar.gz` from
  `https://chroma-onnx-models.s3.amazonaws.com/...` via a bare `httpx` GET
  (no body, no query params, SHA256-pinned). Fires at server runtime on cache
  miss (not at image build). Embedding inference runs fully in-process — corpus
  text is never in a request. **Fix:** pre-bake the six model files into the
  image / cache mount so `__call__` never opens a socket; hand `DocStore` an
  explicit local `embedding_function`.
- **rust-analyzer binary download — already mitigated (build-time only).**
  `PatchedRustAnalyzer.setup_runtime_dependencies` returns the container's native
  binary (installed via rustup at build time); the GitHub-release download path
  is never reached. Confirmed: download target dir does not exist.

## 5. Latent / unreachable dependencies (checked and cleared)

- **huggingface-hub / hf-xet / fsspec** — transitive via `tokenizers`; the ONNX
  EF loads the tokenizer via `Tokenizer.from_file(local path)`. No
  `from_pretrained` / `hf_hub_download` anywhere. Dead.
- **kubernetes / oauthlib / requests-oauthlib / googleapis-common-protos /
  aiohttp / websocket-client / urllib3** — distributed-mode only; `PersistentClient`
  uses local Rust bindings + `LocalSegmentManager`.
- **HTTP-server stack (uvicorn / starlette / websockets / watchfiles / httptools /
  uvloop / python-multipart)** — chromadb server mode / MCP SSE-HTTP only. MCP
  runs stdio; chromadb is embedded. No `listen()`/`bind()` on the runtime path.
- **OTLP gRPC exporter (+ grpcio)** — imported but never constructed.

First-party dataflow cleared: no module reads `.rs` bodies (only paths/positions
flow to the local rust-analyzer subprocess); the only two subprocess spawns are
`git rev-parse HEAD` (argv-list, no shell, local-only); `core.py`'s `urllib.parse`
is pure `file://` URI string parsing; markdown flows only into on-disk ChromaDB.

## 7. Follow-up gaps (from the completeness critic)

Not blocking, but worth closing for a definitive audit:

1. **The compiled Rust backend (`chromadb_rust_bindings*.abi3.so`) was not
   inspected** — it *is* the PersistentClient engine. Check with `nm -D`/`strings`
   for `reqwest|hyper|https://|socket` and `ldd`/`readelf -d` for linked network
   libs; confirm sqlite-persistence-only.
2. **`aiohttp`'s runtime importer is unidentified** — confirm what pulls it in and
   that it's unreachable at runtime.
3. **`onnxruntime` native telemetry not examined** — `strings` the `.so` for
   telemetry endpoints; confirm the ETW provider is Windows-only / no HTTP sink.
4. **Telemetry re-enable surface** — ChromaDB `Settings` also honors
   `chroma_product_telemetry_impl` / `chroma_telemetry_impl` (dotted class path)
   and `chroma_otel_*` via `CHROMA_*` env; confirm the hardcoded
   `anonymized_telemetry=False` can't be bypassed by an env-supplied impl class.
5. **Lock-level invariant** — add a CI/lock guard asserting `posthog` (and
   sentry/bugsnag/etc., already absent) never enters the resolved tree.

**Verified clean:** `sentry_sdk`/`bugsnag`/`rollbar`/`mixpanel`/`segment`/`scarf`
/`posthog` are all absent from `uv.lock` and not importable — no crash/error-
reporting egress dependency exists.
